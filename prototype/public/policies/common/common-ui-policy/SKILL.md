# 🎯 목적

이 스킬은 **공통 UI 정책**을 기반으로 화면별 상세 기획문서를 자동 생성한다.

---

# 📋 정책 파일 참조

공통 UI 정책의 상세 내용은 다음 파일을 참조한다:
- **위치**: `prototype/policy/common/common-ui-policy.md`
- **내용**: 리스트 화면, 추가/수정 팝업, 금액 포맷, 서버 에러 등 공통 정책

---

# 🚀 사용 방법

```
/common-ui-policy [화면명] [저장 위치]

예시:
/common-ui-policy 임직원 관리 리스트 policy/HR/
```

**입력 요구사항**과 함께 실행하면, 공통 정책 + 화면별 요구사항을 결합하여 상세 기획문서를 생성한다.

---

# 📝 기획문서 생성 규칙

## 충돌 처리 규칙(중요)
- 화면별 정책이 공통 정책과 다르면 화면별 정책을 우선 적용한다.
- 단, 공통 정책에서 무엇이 변경/무시되었는지 "충돌 지점"을 출력에 명시한다.

---

## 산출물 규칙(Claude가 화면 구현/문서화 시)
- 구현 또는 화면정의서 작성 시, 공통 정책 항목이 적용되었음을 체크리스트 형태로 포함한다.
- 공통 정책에 의해 자동으로 들어간 항목(예: 페이지 사이즈, 엑셀 다운로드 범위, confirm 문구)은 누락하지 않는다.
- 화면별 추가 요구사항이 주어지면 공통 정책 + 추가 요구사항을 합쳐 최종 정책을 제시한다.

## 질문 및 확인 규칙 (중요)

- 공통 정책 또는 화면별 요구사항만으로 구현/설계가 명확하지 않은 경우,
  임의로 추측하여 구현하지 않는다.
- 불명확하거나 누락된 사항은 반드시 "질문 목록"으로 정리하여 제시한다.

### 질문이 필요한 대표 사례
아래 항목 중 하나라도 명확하지 않으면 질문 대상으로 분류한다.

1. 리스트 화면
   - 검색항목이 명시되지 않은 경우
   - 기본 정렬이 최근 등록 순이 아닌 경우의 기준
   - 삭제의 의미(물리삭제/논리삭제/데이터만 초기화 등)
   - 엑셀 업로드/다운로드 시 부분 성공 vs 전체 실패 정책
   - 행 클릭 시 동작이 수정 팝업 외 다른 동작일 가능성

2. 추가/수정 팝업
   - 필수/선택 입력 여부가 명확하지 않은 필드
   - 유효성 규칙(길이, 범위, 중복 체크 등)이 정의되지 않은 경우
   - 저장 실패 시 UX 처리 방식(에러 문구/토스트/팝업 유지 여부)
   - 삭제 가능 여부 조건이 정의되지 않은 경우
   - 삭제 후 이동할 리스트 화면이 존재하지 않거나 불명확한 경우

3. 공통 정책과 화면별 정책의 충돌
   - 공통 정책과 상이한 동작이 요구될 경우
   - 어떤 정책을 우선 적용해야 하는지 명확하지 않은 경우

4. 금액/숫자 입력
   - 소수점 허용 여부가 명시되지 않은 경우
   - 음수 허용 여부가 명시되지 않은 경우
   - 반올림/절사 규칙이 필요한 경우

---

### 질문 산출 방식(강제)
- 질문은 별도의 섹션 또는 별도 파일로 정리한다.
- 질문은 아래 형식을 따른다.

Q1. [화면/기능명]
- 확인 필요 내용:
- 현재 정보 기준 가정 시 문제점:
- 선택 가능한 옵션(있을 경우):

### 질문 예시
- 삭제 기능은 물리삭제인가요, 논리삭제인가요?
- 엑셀 업로드 시 한 행이라도 오류가 발생하면 전체 롤백되나요?
- Enter 키 입력 시 검색을 실행해야 하나요?
- 팝업에서 ESC 키로 닫기를 허용하나요?

### 질문 완료 시 처리 규칙
- 모든 질문에 대한 답변이 완료되어 정책에 반영되면, 정책서에서 질문 섹션을 삭제한다.
- 질문 섹션이 남아있는 정책서는 미결 사항이 있는 것으로 간주한다.
- 질문이 처음부터 없는 경우(모든 요구사항이 명확한 경우)에도 질문 섹션을 생성하지 않는다.
